# pytest-django-capquery
Тесты django-приложений без лишних обращений к БД

## Описание проблемы
Эффективная разработка требует эффективного тестирования. Мой стек: django + DRF + srf_spectacular, это худшая комбинация, для написания ээфективных тестов API.

Минимальное view в таком стеке:
``` python
class UserViewSet(mixins.CreateModelMixin, viewsets.GenericViewSet):
    queryset = User.objects.all()
    serializer_class = UserSerializer
```
Проблема в том, что при тестировании api на основе такого view, тест проверяет не только логику view, но все middleware и взаимодействия с БД.
Если middleware проходит достаточно быстро, то взаимодействие с БД - медленное. Как следствие, когда количество тестов приближается к 1000, то время выполнения тестов становится некомфортным.

## Расмотренные решения
### 1. Sqlite
Очевидное решение - использовать inmemory sqlite. Для меня это не сработало. В версии django 2.2 sqlite не поддерживал JsonField и distinct

### 2. DDD
Во взрослых проектах, слой получения данных выделен. Это позволяет его подменять во время тестирования. Другими словами, можно сделать так, чтобы при работе использовался слой получения данных который обращается к БД, а при тестировании точно такой же слой получения данных, но который использует память вместо БД. 

Перед тем как использовать DDD в рабочем проекте, я решил создать свой пет-проект на основе DDD. Оказалось, что:
* во-первых, это очень сложно сделать
* во-вторых, сложно поддерживать, каким бы страным это не казалось, 
* в-третьих, каждый член команды должен иметь такое же видение архитектуры коды, как и все остальные. 

Дальше пет-проекта дело не пошло.

### 3. Правильные сервисы
Большинство разработчиков django согласны с тем, логика должна выносится к сервисы, обслуживающий слой. Данные полученные на вход (API, cli) передаётся в сервис, а он выполняются всю работу. Это похоже на DDD, но значительно проще.
В идеале, сервис не должен напрямую обращаться к БД. назовём такой сервис правильным. Для правильных сервисов, можно было написать аналог ModelSerializer. Такой сериализатор передавал очищенные данные не в модель, а в сервис. А сервис, или использовал бы БД для приложения, или память в тестировании.

К сожалению, есть `filter_backends` на который завязан `drf_pectacular`. 

Попытки сделать так, чтобы Django Rest Framework мог не обращаться к БД во время тестирования, превращает его в [Django Modern Rest](https://github.com/wemake-services/django-modern-rest) Поэтому, правильные сервисы хорошо сочетаются с DMR, но не DRF. Поэтому, если у меня будет DMR-проект, то я обязательно буду использовать правильные сервисы.


## О проекте
Перечисленные выше способы предполагают, что работу с БД нужно выносить в отдельный слой, который можно подменять в тестах, т.е. требуется переписывание кода.
После долгих рассуждений я пришёл к выводу, что можно сделать сделать эффективные тесты не меняя ни одной строчки кода в существующих проектах.
Не нужно проверять, что БД работает. Нужно проверить, что для БД готовится правильный запрос, и что если использовать данные, которые вернул бы БД, API вернёт правильный ответ.
_Любопытный факт: только что (2026-05-31) приложение github предложило посмотреть [awesome-pytest](https://github.com/augustogoulart/awesome-pytest), а в нём появился проект [pytest-capquery](https://github.com/fmartins/pytest-capquery/), который был опубликован в pypi всего 2 месяца назад (2026-04-02). Этот проект делает ровно то, что я хочу сделать, но делает это для SqlAlchemy. Название того проекта мне кажется удачным, поэтому для своего проекта возьму похожее название_

Так как я никогда не писал библиотек, то мне нужны ориентиры. Я планирую взять за основу следующие библиотеки:
* [Django Modern Rest](https://github.com/wemake-services/django-modern-rest) - как пример того, как должны проверяться код и документация
* [pytest-capquery](https://github.com/fmartins/pytest-capquery/) - как пример фикстур и команд
* [pytest-testmon](https://github.com/tarpas/pytest-testmon) - как пример сохранения данных в sqlite
* [pytest-insta](https://github.com/vberlier/pytest-insta) - как пример хранения текстовых данных между тестами
* [django-cacheops](https://github.com/Suor/django-cacheops) - как пример перехвата запросов и ответов django orm

Моя библиотека должна делать следующее:
* Перехватывать запрос к БД и ответы от БД, как в django-cacheops
* В случае успешного выполнения теста, перехваченные данные записываются в текстовые файлы, как pytest-insta
* При повторном запуске тестирования. данные из всех тестовых собираются в inmemory sqlite, как в pytest-testmon
* При выполнении теста, используются закешированные с прошлого раза данные.
* Если тест падает, то текстовые данные удаляются, вынуждая данные обновляться
